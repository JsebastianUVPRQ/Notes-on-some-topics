# Activity: Building an intelligent blog application

## Objective

In this lab, you will practice test-driven development (TDD) by building a simple blog application using Flask. You will start with a pre-written test suite and then implement the application code to make the tests pass. This exercise will reinforce the principles of TDD and give you hands-on experience with integration testing in a web development context.

### Tips for completing this lab

As you navigate this lab, keep the following tips in mind:

- The instructions for this lab are separated into multiple steps in this window. You can use the dropdown menu at the top or the next and back buttons at the bottom to navigate through the instructions.
    
- You will need to follow the setup instructions in **Step 1.2** each time the lab is rebooted. 
    
- You can access the terminal in VS Code by using **ctrl-`** or by navigating to **View -> Terminal**.
    

## Scenario

You are a Python developer learning about test-driven development. You want to experience the TDD workflow by building a blog application where users can create and view posts. ## 2. Fix initialization test (5 minutes)

### Step 2.1: Examine test_posts_initialized_as_empty_list 

While you will not modify the tests, you can open **test_app.py** from the tests folder to see what each test does. The first test, test_posts_initialized_as_empty_list, checks that the posts variable in your **app.py** file starts as an empty list. The posts variable stores all the blog posts in your application. The test makes sure it is completely empty when the application starts. This is important because you don't want any old or unexpected posts showing up when you haven't added any yet.

The test has two main parts:

**Import:** It imports the posts variable from your **app.py** file to enable the posts to be examined.

**Check:** It checks if the posts variable is equal to an empty list ([]). It ensures that your blog application starts fresh, with no leftover posts from previous uses. This test is like a safety check to make sure everything is set up correctly before you start adding posts to your blog. It's a small but important step in building a reliable application. 

### Step 2.2: Correct the posts initialization

Open the **app.py** file. Locate the line that says posts = "". This variable should hold a list of posts. It should contain a list of values. You will initialize it to be an empty list. Change the line to:

posts = []

Save the file.

### Step 2.3: Re-run the tests

## 3. Implement the /posts endpoint (GET) (15 minutes)

Now that you've fixed the initialization of the posts variable, it's time to implement some actual functionality in your blog application.  But first, let's take a look at the next test we need to pass: test_get_posts.

### Step 3.1: Understanding test_get_posts 

This test simulates a user's browser requesting a list of all blog posts. It checks that your application responds correctly to this request. Specifically, it verifies two things:

**Status code:** It checks that the server responds with a 200 OK status code. This code means that the request was successful, and the server is sending back the requested data.

**Empty list:** Since there are no posts created yet, the test expects the server to return an empty list ([]).

### Step 3.2: Implement the get_posts function

You need to implement the /posts endpoint to handle GET requests (requests for retrieving data). This endpoint should return a list of all blog posts in JSON format.  JSON is a common way to send data over the web.

The get_posts function in **app.py** already has a decoration for a GET at the /posts URL. The function should be modified to return the posts, encoded in JSON format. Modify the function to return the posts variable encoded in JSON format. Hint: The required library (jsonify) has been imported.

### Step 4.3: Handle valid data

Now that invalid data has been handled, and the test_create_post_invalid_data test is passing, you have two more tests. Both of these tests assume that the data coming in is valid, and you'll need to focus on ensuring the /posts endpoint properly handles this "happy path."

test_create_post: This test checks that your endpoint can successfully create a new blog post when given correct data. It verifies that the endpoint accepts a POST request with a title and content, returns a 201 Created status code (which signals success), and sends back the newly created post in JSON format.

test_create_post_mock_posts: This test might seem a bit more complex, but it's a powerful way to make sure your code is working correctly. It uses a technique called "mocking."  Essentially, it creates a temporary, fake version of the posts list. This allows the test to isolate the create_post function and check if it's correctly adding new posts to the list, without worrying about any other parts of the application. This isolation helps pinpoint exactly where any issues might be. For example, in this example, if the posts list was declared incorrectly, this test would still pass, while the test_create_post test would fail. In a real-world situation, this could test the functionality of create_post without having to concern yourself with whether the database operations work.

To pass these tests, you'll need to write code that:

- Creates a dictionary called new_post to store the new post's data. This dictionary should have the following keys:
    
    - 'id': An integer representing the post's ID. You can generate this using len(posts) + 1 (the current length of the posts list plus 1).
        
    - 'title': A string containing the post's title, extracted from the incoming request data.
        
    - 'content': A string containing the post's content, also extracted from the incoming request data.
        
- Appends the new_post dictionary to the posts list.
    
- Returns the new_post dictionary as JSON with a 201 Created status code. Use jsonify(new_post) to convert the dictionary to JSON. The complete return statement should look similar to: return jsonify(new_post), 201
    

Follow the comments in the create_post function to guide you through these steps.
